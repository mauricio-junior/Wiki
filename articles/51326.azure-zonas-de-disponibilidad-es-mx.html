INTRODUCCIÓN


 Las Zonas de disponibilidad de Azure están diseñadas para reducir un único punto de falla dado un escenario IaaS específico, principalmente descrito como:

"Ubicaciones aisladas de fallas dentro de una región Azure, que proporcionan energía redundante, refrigeración y redes. Las Zonas de disponibilidad permiten a los clientes ejecutar aplicaciones de misión crítica con mayor disponibilidad y tolerancia a fallas a las fallas del centro de datos ".

CARACTERÍSTICAS PRINCIPALES DE LAS ZONAS DE DISPONIBILIDAD DE AZURE:

  • Las zonas de disponibilidad (AZ) son ubicaciones físicamente separadas dentro de una región de azure
  • Cada AZ tiene energía, red y enfriamiento independientes
  • Las ubicaciones AZ se eligen según una evaluación de riesgos por región
  • AZ están diseñados para reducir puntos únicos de falla en la plataforma

 

¿CÓMO PUEDEN AYUDARTE LAS ZONAS DE DISPONIBILIDAD DE AZURE ?

Analicemos 3 enfoques en los que las zonas de disponibilidad pueden ayudarlo a aumentar su resiliencia:

  • Alta disponibilidad y recuperación de desastres
  • Baja latencia para la replicación síncrona
  • Protección contra la falla del nivel del centro de datos

 

Alta disponibilidad y recuperación de desastres:

Soporte nativo de HA- Alta Disponibilidad  y Recuperación de Desastres a través de las zonas de disponibilidad y regiones pares. Cada región habilitada con Zonas de DIsponibilidad tiene un mínimo de tres zonas. Todas las regiones tienen una región par que proporciona replicación de datos con resiliencia de datos.

 

Baja latencia para la replicación síncrona:

Las zonas están lo suficientemente cerca una de la otra para la replicación sincrónica, lo que significa una alta disponibilidad para las aplicaciones y una mínima pérdida de datos.

 

 

Protección contra fallas en el centro de datos:

Cada zona se separa físicamente con energía, red y enfriamiento independientes y se separa lógicamente a través de servicios de zona aislada.

Consideraciones:

  • Cada zona consta de uno o más centros de datos dentro de una región
  • Puede equilibrar la carga en todas sus máquinas virtuales implementadas en zonas de disponibilidad, habilitando escenarios con interfaces zonal y balanceo de carga entre zonas para el back-end.
  • Para las regiones que no admiten zonas de disponibilidad, los conjuntos de disponibilidad aún se deben aprovechar para HA (y para obtener SLA).
  • Las zonas de disponibilidad protegen contra fallas locales y cubre un conjunto más amplio. El mantenimiento planificado acaba de lanzarse como un servicio y puede ser un sustituto de los dominios de actualización.
  • Los conjuntos de disponibilidad y las zonas de disponibilidad no son compatibles al mismo tiempo.
  • En el caso de Always On, el equipo de ingeniería de SQL recomienda distribuir sus nodos por igual en las tres zonas para obtener la mayor disponibilidad.
